## Next.js를 선택한 이유

### 직접 배포 방식

개발 블로그를 만들기 전, 티스토리나 velog같은 서비스 말고 직접 배포하는 방식을 선택하고 싶었다.
E-Book도 플랫폼이 망하면 읽지 못하는데 블로그 서비스라고 다를리 없으니 말이다.

### Jekyll에서 Next.js로

처음에는 개발 블로그를 Github Pages로 배포하는 방법을 선택했고, 이를 위해 Jekyll을 사용했다. 
가장 찾기 쉬웠던 [jekyll-theme-chirpy](https://github.com/cotes2020/jekyll-theme-chirpy) 템플릿을 이용해 블로그를 배포했다. 
하지만 템플릿 디자인이 내 취향과 맞지 않았고, 커스터마이징 과정도 어려웠다.

그래서 새로운 테마인 [jekyll-theme-satellite](https://github.com/byanko55/jekyll-theme-satellite)를 선택해 재배포했다. 
하지만 Jekyll 자체가 Ruby 환경 기반이라 환경 설정이 복잡했고, 내가 원하는 기능을 추가하기 위해서 새롭게 배워야 하는 부담이 컸다. 
그래서 친숙한 기술 스택으로 블로그를 배포하는 방법을 찾기 시작했다.

자바스크립트는 어느정도 간단하게 사용할 줄 알고 있었기 때문에,
자바스크립트 기반의 정적 사이트 생성기들을 비교하면서 Gatsby, Astro, Next.js를 알게 되었다.

- **Gatsby**는 정적 사이트에 최적화되어있지만 데이터를 가져올 때 GraphQL을 사용해 학습곡선이 높아보였다.
- **Astro**는 심플하고 빠른 정적 사이트 생성을 지원했지만, 상대적으로 사용자가 적었다.
- **Next.js**는 원래부터 알고 있던 프레임워크였고, SSG도 지원하고 있었다.

그래서, 사용자가 가장 많아 정보가 많을 것 같은 **Next.js**로 선택하였다.

## 프로젝트 시작 전 준비

블로그 디자인은 기존 Jekyll 블로그에서 가장 마음에 들었던 템플릿인 [jekyll-theme-satellite Demo](https://byanko55.github.io/)을 참고해 디자인했다.

### 사용한 스택 / 라이브러리
* Typescript
* Next.js 15.x.x
* TailwindCSS
* Markdown (MDX)
* pagefind : 정적 사이트 검색 라이브러리

---

## 개발 과정 중 어려웠던 점

프론트엔드 기술에 대해 아는 것이 없었던 것이 제일 힘들었다.
블로그를 만들면서 프론트엔드의 기초는 다지게 됐다.

Next.js의 App Router가 Pages Router로 대체되는 시점으로 보였다. 여러 자료에서 정보가 혼재되어 있었고,
초기 개발에서 기존 Pages Router 방식의 자료들을 따라하다가 잘못된 방향으로 접근한 적도 있다.
하지만 공식문서에서 App router에 관한 정보가 아주 상세히 되어있어 공식문서에서 많이 도움을 얻었다.

블로그 개발 중에서는 디자인이 가장 어려웠고, 시간을 가장 많이 잡아먹었다. 
블로그 글 로드와 mdx 통합에는 큰 어려움이 없었지만 원하는 블로그 템플릿의 CSS를 분석하고 Tailwind로 구현하는 시간이 상당히 소요되었다.

## 향후 계획

**프로젝트 결과 정리**
현재 만들어진 블로그 페이지는 아주 만족스럽지만, 더 개선할 여지가 많다고 느껴진다.
원하는 기능 및 동작에는 문제가 없지만, 세부 디자인이나 사소한 부분들이 부족하다.

**추가 수정 계획**
- Pagefind SearchAPI 통합 : Pagefind DefaultUI를 사용하여 구현했기 때문에 상세한 디자인이 불가능했다.
- 로딩 로직 리팩토링 : 블로그 글을 빌드할때 로드하는 과정이 비효율적으로 느껴져 리팩토링을 하고 싶다. 코드 읽기도 어려운 것 같다.
- tailwind 컨벤션 : 현재 tailwind는 누더기 골렘마냥 마구잡이로 작성되어있다. 유지보수를 한다면 꼭 수정해야할 부분이다.
- TOC 하이라이트 기능 개선 : 동작은 하지만, 한 화면에 여러 헤딩이 잡힐 경우 조금 이상하게 동작한다.
